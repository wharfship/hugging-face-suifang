# Vercel 新手使用指南

## 1. 这件事本质上是在做什么

你现在的前端项目之所以只能在自己电脑上打开，是因为它目前运行在本地环境里。常见情况有两种：

- 浏览器直接读取你电脑里的静态文件
- 你在本地启动了开发服务器，比如 `localhost:3000`、`localhost:5173`

这里的 `localhost` 只代表“你自己的电脑”。别人访问他们自己的 `localhost`，访问到的是他们自己的机器，不是你的机器，所以他们当然打不开你的网页。

要让别人也能访问，你需要做的是：

1. 把前端代码上传到一个公网可访问的平台
2. 让平台帮你构建并发布网站
3. 获得一个公开网址
4. 把这个网址发给别人

Vercel 就是专门做这件事的平台。

## 2. Vercel 到底是什么

Vercel 不是传统意义上“你自己去租服务器、装系统、配 Nginx、配 HTTPS”的那种方案。

它更像一个前端部署平台：

- 你把代码交给它
- 它自动识别你的前端框架
- 它在云端帮你安装依赖并构建项目
- 它把生成好的页面文件部署到 CDN
- 它自动给你一个网址
- 它还能自动配置 HTTPS

对于新手来说，Vercel 的价值在于：省掉了大部分服务器配置工作。

## 3. Vercel 的工作原理

你可以把 Vercel 的流程理解成下面这几步：

### 第一步：你提供代码

通常有两种方式：

- 把项目放到 GitHub，然后让 Vercel 连接 GitHub
- 在本地使用 Vercel CLI 直接上传项目

### 第二步：Vercel 在云端构建项目

比如你的项目是 React、Vue、Vite、Next.js，Vercel 会根据配置执行类似下面的流程：

- 安装依赖
- 运行构建命令，比如 `npm run build`
- 生成最终可部署的文件

### 第三步：Vercel 发布构建结果

构建成功后，Vercel 会把生成出来的文件放到可公开访问的网络节点上。别人访问网址时，拿到的是这套已经构建好的前端文件。

### 第四步：生成公开网址

部署成功后，你会得到一个类似下面的地址：

```text
your-project.vercel.app
```

这个地址通常就可以直接给别人访问。

### 第五步：后续自动更新

如果你使用的是 GitHub 方式，以后你每次提交新代码并推送，Vercel 都可以自动重新部署。

## 4. 新手必须先明白的一个区别

很多人第一次部署前端时，会误以为“只要把网页传上去就行了”。其实你需要先判断你的项目到底是哪一种：

### 情况 A：纯前端项目

例如：

- 纯 HTML/CSS/JS
- React/Vue/Vite 的纯页面项目
- 页面只展示内容，不依赖你本地后端

这种项目通常最适合直接部署到 Vercel。

### 情况 B：前端 + 本地后端

例如你的前端代码里写了：

```text
http://localhost:3000
http://127.0.0.1:5000
```

或者依赖：

- 你电脑上跑着的 Python/Node/Java 后端
- 你本地数据库
- 你电脑上的文件目录

那就要注意：即使前端部署到 Vercel 了，别人打开页面后，这些本地接口依然访问不到。

也就是说：

- 前端部署不等于后端也自动部署
- 网页能打开不等于功能一定完整可用

## 5. Vercel 适不适合你

如果你目前的目标只是：

- 让别人能打开你的前端页面
- 先把静态网站或前端项目发布出去
- 不想自己折腾服务器环境

那么 Vercel 很适合你。

如果你的项目后面还有复杂后端、数据库、文件上传、长期任务等内容，也不是不能用 Vercel，但学习成本会逐步上升。

对于新手，建议顺序是：

1. 先学会把前端页面上线
2. 再处理接口和后端
3. 最后再考虑域名、优化、监控这些进阶问题

## 6. Vercel 的价格怎么理解

截至 2026-03-29，Vercel 官方价格页显示：

- `Hobby`：`Free forever`
- `Pro`：`$20/month + additional usage`

对新手个人项目来说，通常先用 `Hobby` 就够了。也就是说，它不是你理解中的“便宜软件”，更准确地说是“个人项目入门成本很低，甚至可以免费开始”。

## 7. 最推荐新手的方式：GitHub + Vercel 后台部署

这是最稳、最适合长期使用的办法。

### 7.1 你需要提前准备什么

先准备这几样：

- 一个本地前端项目
- 安装好 Node.js
- 项目在本地能正常运行
- 一个 GitHub 账号
- 一个 Vercel 账号

### 7.2 先在本地确认项目没问题

在项目目录里先做这两件事：

1. 安装依赖
2. 测试项目是否可以构建

常见命令：

```bash
npm install
npm run build
```

如果 `npm run build` 都过不了，就不要急着部署，因为 Vercel 云端也大概率会失败。

### 7.3 把项目上传到 GitHub

如果你已经会 Git，就把项目推到 GitHub 仓库。

如果你还不会，可以先记住目标：

- 在 GitHub 创建一个仓库
- 把你的前端项目代码上传上去

只要代码在 GitHub 上，后面连接 Vercel 就很方便。

### 7.4 注册并登录 Vercel

打开：

`https://vercel.com`

最省事的方式是直接用 GitHub 账号登录。

### 7.5 导入你的 GitHub 仓库

登录后：

1. 点击 `New Project`
2. 选择 `Import Git Repository`
3. 授权 Vercel 访问你的 GitHub 仓库
4. 选择你的项目仓库

### 7.6 检查自动识别到的配置

Vercel 往往会自动识别你的项目类型，例如：

- Vite
- React
- Next.js
- Vue

它通常还会自动填写：

- `Framework Preset`
- `Build Command`
- `Output Directory`

你需要重点看两项：

- 构建命令对不对
- 输出目录对不对

常见例子：

- Vite 常见输出目录：`dist`
- Create React App 常见输出目录：`build`

### 7.7 如果项目用到了环境变量，先配置

很多项目本地会有 `.env` 文件。这些变量不会自动从你电脑同步到 Vercel。

你需要在 Vercel 后台的项目设置里手动填写，例如：

- `VITE_API_URL`
- `NEXT_PUBLIC_API_URL`
- 第三方服务 key

注意：

- 不要把敏感信息直接硬编码到前端代码里
- 不要以为本地 `.env` 会自动跟着部署过去

### 7.8 点击 Deploy

确认配置后，点击 `Deploy`。

部署成功后，Vercel 会给你一个网址，通常类似：

```text
your-project.vercel.app
```

这时别人就可以通过这个网址访问你的页面。

### 7.9 以后怎么更新

如果你是 GitHub 接入方式，以后流程一般是：

1. 你在本地修改代码
2. 提交并推送到 GitHub
3. Vercel 自动检测到新提交
4. 自动重新部署

这就是为什么我更推荐新手用 GitHub 方式，而不是每次手动上传。

## 8. 第二种方式：使用 Vercel CLI 从本地直接部署

如果你暂时不想先学 GitHub，也可以直接从本地部署。

### 8.1 先安装 Node.js

如果你的电脑还没有 Node.js，先安装它。建议使用较新的 LTS 版本。

### 8.2 安装 Vercel CLI

打开终端后执行：

```bash
npm i -g vercel
```

### 8.3 登录

```bash
vercel login
```

按提示完成登录。

### 8.4 进入你的项目目录

```bash
cd 你的项目目录
```

### 8.5 开始部署

```bash
vercel
```

第一次运行时，CLI 会问你一些问题，例如：

- 是否把当前目录链接为一个项目
- 项目名称是什么
- 使用什么框架
- 输出目录是什么

如果只是想先试一版，`vercel` 通常会生成一个预览部署。如果你想直接发正式版本，可以使用：

```bash
vercel --prod
```

## 9. 什么叫“构建命令”和“输出目录”

这两个词是新手第一次部署时最容易卡住的地方。

### 构建命令

构建命令就是：如何把源码变成可上线文件。

比如很多前端项目会使用：

```bash
npm run build
```

### 输出目录

输出目录就是：构建完成后，最终生成文件放在哪个文件夹里。

例如：

- `dist`
- `build`

Vercel 需要知道最终应该发布哪个文件夹。

如果你的项目是非常简单的纯静态页面，有时甚至不需要构建命令，直接发布静态文件也可以。

## 10. Preview 和 Production 是什么

Vercel 里经常会看到两个概念：

### Preview

预览版本。适合先自己看效果、测试功能、给少数人试用。

### Production

正式版本。这是你真正对外公开的主要网址。

新手可以简单理解为：

- `Preview`：测试版
- `Production`：正式版

## 11. 自定义域名怎么做

如果你不满足于 `xxx.vercel.app`，想绑定自己的域名，比如：

```text
www.yourname.com
```

也可以。

### 基本流程

1. 先把项目成功部署到 Vercel
2. 在项目设置中找到 `Domains`
3. 添加你的域名
4. 按照 Vercel 提示去域名服务商后台修改 DNS
5. 等待 DNS 生效
6. Vercel 自动签发 HTTPS 证书

对新手来说，域名这一步不是第一优先级。建议先把 `vercel.app` 地址跑通，再做域名绑定。

## 12. 新手最常见的坑

### 坑 1：项目本地能打开，但部署失败

原因通常是：

- 本地只是 `npm run dev` 能跑
- 但 `npm run build` 其实报错

解决思路：

- 先在本地跑通 `npm run build`
- 再去部署

### 坑 2：网页打开了，但接口请求全失败

常见原因：

- 前端请求的是 `localhost`
- 后端还在你自己电脑上

解决思路：

- 把后端也部署到公网
- 或者先把前端改成可访问的线上接口地址

### 坑 3：本地正常，Vercel 上报环境变量错误

原因通常是：

- 本地 `.env` 有值
- Vercel 没有配置同样的变量

解决思路：

- 在 Vercel 项目设置中补齐环境变量

### 坑 4：改了代码，线上没更新

可能原因：

- 你只改了本地，没有推送到 GitHub
- 你部署的是 Preview 版本，不是 Production

### 坑 5：以为 Vercel 会自动帮你处理数据库

不会。前端、后端、数据库是不同层的问题。Vercel 能很好地解决前端部署问题，但不代表所有后端基础设施都自动完成。

## 13. 给新手的推荐学习顺序

建议你按下面这个顺序走：

1. 先确认自己的项目只是前端，还是还依赖本地后端
2. 先在本地跑通 `npm run build`
3. 使用 GitHub + Vercel 部署出一个 `vercel.app` 链接
4. 用别人的电脑或手机测试能否打开
5. 如果页面能打开，再检查接口功能是否正常
6. 最后再考虑自定义域名

## 14. 最短实操路线

如果你现在就想尽快做出来，可以直接照这个流程：

1. 本地项目先运行 `npm run build`
2. 把项目上传到 GitHub
3. 用 GitHub 登录 Vercel
4. 点击 `New Project`
5. 导入你的仓库
6. 检查框架、构建命令、输出目录
7. 配置环境变量
8. 点击 `Deploy`
9. 拿到 `xxx.vercel.app` 地址
10. 发给别人测试

## 16. 一句话总结

如果你只是想让别人能打开你的本地前端，最简单的理解是：

把代码交给 Vercel，Vercel 帮你构建并发布成一个公开网址。

GitHub 上传代码 -> Vercel 导入仓库 -> 点击部署 -> 拿到网址发给别人。
